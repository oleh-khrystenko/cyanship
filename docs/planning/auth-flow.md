## Принципи

1. **Один email = один акаунт** — незалежно від провайдера (Google, magic link, пароль). Якщо юзер зареєструвався через Google і потім вводить той самий email — це той самий акаунт.
2. **Пароль — опціональний** — юзер завжди може входити через OAuth або magic link без пароля. Пароль — додатковий рівень зручності та безпеки.
3. **Progressive disclosure** — спочатку тільки поле email, далі контекстно показується або поле пароля, або повідомлення про magic link.
4. **Boilerplate-friendly** — ключові параметри (кількість спроб, довжина пароля, тощо) мають бути конфігурованими, а не захардкодженими.

## Сторінка авторизації

### UI елементи

```
┌─────────────────────────────────────────┐
│                                         │
│            Вхід до LucidKit             │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │      Увійти з Google              │  │
│  └───────────────────────────────────┘  │
│                                         │
│           ──── або ────                 │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │  Email                            │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │           Продовжити              │  │
│  └───────────────────────────────────┘  │
│                                         │
└─────────────────────────────────────────┘
```

### Поведінка

1. Юзер вводить email і натискає "Продовжити"
2. Показується прелоадер
3. Backend перевіряє: чи існує юзер і чи має пароль
4. В залежності від результату — один з трьох сценаріїв (див. нижче)

> **Безпека:** Endpoint `check-email` захищається rate limit по зв'язці IP + email (не лише email), щоб запобігти DoS-блокуванню легітимних юзерів (див. [Захист від brute force](#захист-від-brute-force)).

## Сценарії email-авторизації

### Сценарій A: Юзер існує + має пароль

**Тригер:** Backend повертає `{ hasPassword: true }`

**UI:** Під полем email з'являється поле пароля (з toggle show/hide) + кнопка "Увійти" + посилання "Забув пароль?" + посилання "Змінити email" для повернення до кроку вводу email.

```
┌─────────────────────────────────────────┐
│  ┌───────────────────────────────────┐  │
│  │  email@example.com     [Змінити] │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │  Пароль                      [👁] │  │
│  └───────────────────────────────────┘  │
│              Забув пароль?              │
│  ┌───────────────────────────────────┐  │
│  │             Увійти                │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

- **Show/hide toggle:** Іконка ока у полі пароля перемикає видимість (замість окремого поля "Підтвердити пароль" — сучасний підхід, менше фрустрації).
- **"Змінити":** Повертає на крок вводу email, щоб юзер не натискав Back у браузері.

**Після успішного входу:** Redirect на головну сторінку (`/`), НЕ на профіль.

---

### Сценарій B: Юзер існує + НЕ має пароля

**Тригер:** Backend повертає `{ hasPassword: false, isNewUser: false }`

**UI:** Прелоадер зникає, з'являється повідомлення:

```
┌─────────────────────────────────────────┐
│                                         │
│        Перевірте вашу пошту             │
│                                         │
│  Ми надіслали посилання для входу на    │
│  email@example.com                      │
│                                         │
│  Не отримали листа? Перевірте спам      │
│  або спробуйте ще раз.                  │
│                                         │
└─────────────────────────────────────────┘
```

**Email:** Лист з текстом, що відповідає ситуації — юзер вже зареєстрований, ми надсилаємо посилання для входу та можливість встановити пароль.

---

### Сценарій C: Юзер НЕ існує (новий)

**Тригер:** Backend повертає `{ hasPassword: false, isNewUser: true }`

**UI:** Ідентичний сценарію B (юзер бачить "Перевірте вашу пошту").

**Email:** Лист з текстом про реєстрацію — вітання, magic link для активації акаунту та заповнення профілю.

> **Важливо:** З точки зору UI, сценарії B та C виглядають однаково для юзера. Різниця тільки в тексті email та в стані сторінки профілю після переходу по magic link.

## Magic Link

### Генерація

- Token: 64-byte hex (256-bit), зберігається в Redis
- TTL: 15 хвилин
- Rate limit: 3 запити на email за 15 хвилин
- Одноразовий: після використання видаляється з Redis
- **Anti-spam:** Перед генерацією нового токену перевіряється, чи є в Redis валідний токен для цього email з тим самим purpose, створений менше 1 хвилини тому. Якщо є — новий лист не відправляється, фронту повертається успішна відповідь. Це захищає від подвійного кліку та спаму.

### Повторний клік на magic link (вже використаний)

Коли юзер натискає на magic link вдруге (token вже видалений з Redis):

1. Backend не знаходить token → повертає помилку
2. Redirect на сторінку авторизації
3. Toast-нотифікація: "Посилання вже було використано. Введіть email для отримання нового"

**Обгрунтування:** Magic link одноразовий з міркувань безпеки. Якщо хтось перехопив посилання і юзер вже його використав — зловмисник не зможе увійти. Повідомлення дружнє і одразу пропонує рішення.

### Expired magic link

Коли юзер натискає на magic link після закінчення TTL:

1. Backend не знаходить token (видалений по TTL) → повертає помилку
2. Redirect на сторінку авторизації
3. Toast-нотифікація: "Час дії посилання закінчився. Спробуйте ще раз"

> **Примітка:** Технічно expired та reused magic link повертають однакову помилку (token не знайдено в Redis). Backend не розрізняє ці випадки. Можна використовувати єдине повідомлення: "Посилання недійсне або вже було використано. Введіть email для отримання нового".

## Forgot Password

### Флоу

1. Юзер на сторінці авторизації бачить поле пароля (Сценарій A)
2. Натискає "Забув пароль?"
3. Backend відправляє magic link на email юзера
4. Toast: "Якщо акаунт з цією адресою існує, ми надіслали посилання для зміни пароля"
5. Юзер натискає magic link → потрапляє на сторінку профілю
6. На сторінці профілю поле пароля в режимі редагування (обов'язкове для заповнення)
7. Юзер встановлює новий пароль → зберігає
8. **Після зміни пароля:** Backend ревокує всі інші refresh tokens користувача (`revokeAllUserTokens()`), крім поточної сесії. Юзер на поточному пристрої залишається авторизованим (видається нова пара токенів), на всіх інших — "вилітає" на сторінку логіну

> **Безпека:** Toast на кроці 4 не розкриває чи існує email в базі. Backend завжди повертає однаковий успішний response, незалежно від того чи email знайдений. Це запобігає user enumeration через forgot password.

### Email

Окремий шаблон листа для "Забув пароль" — текст про те, що був запит на зміну пароля та magic link.

## Профіль після magic link

Після успішного переходу по magic link юзер автентифікується (отримує сесію) і потрапляє на сторінку профілю (protected route). Стан сторінки залежить від контексту:

### Новий юзер (Сценарій C → magic link)

Профіль **повністю в режимі редагування:**

| Поле | Стан | Обов'язкове | Примітка |
|------|------|-------------|----------|
| Ім'я | Редагування | Так | Потрібне для звернення в листах та на платформі |
| Прізвище | Редагування | Ні | — |
| Аватар | Редагування | Ні | Можливість завантажити |
| Пароль | Редагування | Ні | Рекомендація: "Для безпеки ваших даних ми рекомендуємо встановити пароль" |

```
┌─────────────────────────────────────────┐
│          Заповніть ваш профіль          │
│                                         │
│  Ім'я *                                 │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│                                         │
│  Прізвище                               │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│                                         │
│  Пароль (рекомендовано)                 │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│  Для безпеки ваших даних ми             │
│  рекомендуємо встановити пароль          │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │           Зберегти                │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### Існуючий юзер без пароля (Сценарій B → magic link)

| Поле | Стан | Обов'язкове | Примітка |
|------|------|-------------|----------|
| Ім'я | Перегляд (не редагується) | — | Вже заповнене |

| Аватар | Перегляд (не редагується) | — | — |
| Пароль | Редагування | Ні | Рекомендація: "Для безпеки ваших даних ми рекомендуємо встановити пароль" |

Профіль **частково в режимі редагування:**

| Поле | Стан | Обов'язкове | Примітка |
|------|------|-------------|----------|
| Ім'я | Перегляд | — | — |
| Прізвище | Перегляд | — | — |
| Аватар | Перегляд | — | — |
| Пароль | Редагування | Так | Юзер прийшов скинути пароль — поле обов'язкове |

## Управління паролем у профілі

На сторінці профілю (доступна з будь-якої точки додатку) є секція "Безпека":

### Юзер без пароля

- Показується кнопка "Встановити пароль"
- Поле: новий пароль (з toggle show/hide)
- Після збереження: пароль з'являється, наступний вхід через email покаже поле пароля

### Юзер з паролем

- Показується кнопка "Змінити пароль"
- Поля: поточний пароль → новий пароль (з toggle show/hide)
- Після збереження: пароль оновлюється
- **Інвалідація сесій:** Після зміни пароля backend ревокує всі інші refresh tokens (`revokeAllUserTokens()`), поточна сесія зберігається (видається нова пара токенів)

### Видалення пароля

- Показується кнопка "Видалити пароль" (або перемикач)
- Підтвердження: "Ви впевнені? Після видалення пароля вхід буде можливий тільки через OAuth або magic link"
- Після видалення: наступний вхід через email відправить magic link замість показу поля пароля

> **Примітка (boilerplate):** Можливість видалення пароля збережена для гнучкості. Конкретний бізнес може приховати цю опцію якщо вважає за потрібне.

## OAuth (Google та інші)

### Поточна реалізація: Google

1. Юзер натискає "Увійти з Google"
2. Redirect на Google consent screen
3. Google callback → `AuthService.handleGoogleAuth()`
4. Якщо юзера немає — створюється (email, profile.name, profile.avatar з Google)
5. Set `bid_refresh` cookie → redirect на callback page
6. Web: `refreshToken()` → `getMe()` → redirect на головну

### Майбутні провайдери

Facebook, Apple та інші додаються за тим самим патерном: кнопка на сторінці авторизації + Passport strategy + callback handler. Принцип "один email = один акаунт" зберігається.

## Account Linking

### Правило

Один email = один акаунт. Провайдер не створює окремий акаунт.

### Сценарії

| Перша реєстрація | Друга дія | Результат |
|------------------|-----------|-----------|
| Google (user@gmail.com) | Email вхід user@gmail.com | Той самий акаунт, magic link для входу (пароля немає) |
| Magic link (user@gmail.com) | Google OAuth з user@gmail.com | Той самий акаунт, Google провайдер додається до акаунту |
| Email + пароль (user@gmail.com) | Google OAuth з user@gmail.com | Той самий акаунт, Google провайдер додається, пароль зберігається |

## Видалення акаунту

### Розташування в UI

Сторінка профілю → окрема секція **"Небезпечна зона"** внизу сторінки (візуально відокремлена, як на GitHub). Кнопка "Видалити акаунт" з деструктивним стилем (червоний).

### Тип видалення: Soft delete + grace period 30 днів

- При запиті на видалення акаунт **деактивується** (поле `deletedAt` встановлюється на поточну дату)
- Юзер автоматично розлогінюється (всі refresh tokens ревокуються)
- Дані зберігаються 30 днів
- Після 30 днів — автоматичне повне видалення (cron job або TTL-based)
- Конфігурована змінна: `ACCOUNT_DELETION_GRACE_DAYS=30`

### Підтвердження видалення

Два варіанти в залежності від наявності пароля:

#### Юзер з паролем

1. Натискає "Видалити акаунт"
2. Модальне вікно з попередженням та поясненням наслідків
3. Вводить пароль для підтвердження
4. Натискає "Підтвердити видалення"
5. Акаунт деактивується → logout → redirect на сторінку авторизації
6. Confirmation email на пошту

```
┌─────────────────────────────────────────┐
│         Видалення акаунту               │
│                                         │
│  Ваш акаунт буде деактивовано.          │
│  Протягом 30 днів ви зможете            │
│  відновити його, увійшовши в систему.    │
│  Після цього всі дані будуть            │
│  видалені безповоротно.                 │
│                                         │
│  Введіть пароль для підтвердження:      │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│                                         │
│  [ Скасувати ]  [ Видалити акаунт ]     │
└─────────────────────────────────────────┘
```

#### Юзер без пароля

1. Натискає "Видалити акаунт"
2. Модальне вікно з попередженням
3. Повідомлення: "Для підтвердження ми надішлемо посилання на вашу пошту"
4. Натискає "Надіслати підтвердження"
5. Magic link на email (purpose: `delete-account`)
6. Юзер натискає посилання в листі → акаунт деактивується → redirect на сторінку авторизації з toast
7. Confirmation email на пошту

```
┌─────────────────────────────────────────┐
│         Видалення акаунту               │
│                                         │
│  Ваш акаунт буде деактивовано.          │
│  Протягом 30 днів ви зможете            │
│  відновити його, увійшовши в систему.    │
│  Після цього всі дані будуть            │
│  видалені безповоротно.                 │
│                                         │
│  Для підтвердження ми надішлемо         │
│  посилання на вашу пошту.               │
│                                         │
│  [ Скасувати ]  [ Надіслати підтвердження ] │
└─────────────────────────────────────────┘
```

### Confirmation email

Після деактивації акаунту на пошту надсилається лист:

- Підтвердження що акаунт деактивовано
- Дата остаточного видалення (поточна дата + 30 днів)
- Інструкція як відновити: "Щоб відновити акаунт, просто увійдіть в систему протягом 30 днів"
- Посилання на сторінку авторизації

### Вхід під час grace period (відновлення акаунту)

Коли юзер з деактивованим акаунтом намагається увійти (будь-яким способом: пароль, magic link, Google OAuth):

1. Backend знаходить юзера з `deletedAt != null`
2. Замість звичайного входу — показується спеціальний екран:

```
┌─────────────────────────────────────────┐
│                                         │
│     Ваш акаунт заплановано до          │
│     видалення                           │
│                                         │
│     Акаунт буде остаточно видалено      │
│     {дата}. Бажаєте відновити           │
│     акаунт?                             │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │         Відновити акаунт          │  │
│  └───────────────────────────────────┘  │
│                                         │
│           Ні, не відновлювати           │
│                                         │
└─────────────────────────────────────────┘
```

3. **"Відновити акаунт"** → `deletedAt` скидається → звичайний вхід → redirect на головну
4. **"Ні, не відновлювати"** → redirect на сторінку авторизації

### Що відбувається під час grace period

- Акаунт існує в базі, але деактивований
- Юзер НЕ може користуватись платформою (всі protected routes недоступні)
- Юзер МОЖЕ увійти — але тільки для відновлення або відхилення
- Всі дані (звіти, платежі, кредити) зберігаються без змін
- Email юзера залишається "зайнятим" — новий акаунт з тим самим email створити не можна

### Автоматичне видалення після grace period

- Cron job (або scheduled task) перевіряє `deletedAt` + 30 днів
- Повне видалення: юзер, пов'язані звіти, платежі, Redis tokens
- Після видалення email звільняється для нової реєстрації

## Захист від brute force

### Невірний пароль — progressive lockout

Замість фіксованого ліміту використовується прогресивне блокування, яке не дратує легітимних юзерів, але ефективно зупиняє ботів:

| Невдалих спроб | Блокування |
|----------------|------------|
| 5              | 1 хвилина  |
| 10             | 5 хвилин   |
| 20             | 15 хвилин  |

- **Ключ:** Redis `login_attempts:{ip}:{email}` — зв'язка IP + email, щоб зловмисник не міг заблокувати вхід для легітимного юзера з іншого IP (DoS prevention)
- **TTL:** Лічильник спроб скидається після 15 хвилин неактивності
- **Повідомлення юзеру:** "Забагато невдалих спроб. Спробуйте через {N} хвилин або скористайтесь посиланням 'Забув пароль?'"
- **Після 20 спроб:** Додатково показувати "Увійти через email-посилання" як основний шлях

### Rate limit для check-email

- **Ключ:** Redis `check_email:{ip}` — per-IP rate limit
- **Ліміт:** 10 запитів на IP за 1 хвилину
- Запобігає масовому перебору email-адрес

### Конфігурація (env або config)

```
AUTH_LOCKOUT_THRESHOLDS=5:1,10:5,20:15   # спроби:хвилини_блоку
AUTH_LOGIN_ATTEMPTS_TTL_MIN=15
AUTH_PASSWORD_MIN_LENGTH=8
AUTH_MAGIC_LINK_TTL_MIN=15
AUTH_MAGIC_LINK_RATE_LIMIT=3
AUTH_MAGIC_LINK_RATE_WINDOW_MIN=15
AUTH_MAGIC_LINK_DEDUP_SEC=60
ACCOUNT_DELETION_GRACE_DAYS=30
```

> **Boilerplate-нотатка:** Progressive lockout — стандартна практика (Google, GitHub). Конфігурується через `AUTH_LOCKOUT_THRESHOLDS` у форматі `attempts:block_minutes`. Ключ `{ip}:{email}` захищає від двох векторів: brute force пароля І DoS через блокування чужого email.

## Вимоги до пароля

### Мінімальні (default)

- Мінімальна довжина: конфігурована змінна (default: `8` символів)
- Без додаткових вимог до складності

### Валідація

- Zod-схема в `packages/types` — single source of truth
- Перевірка на frontend (UX) та backend (безпека)
- Повідомлення: "Пароль має містити щонайменше 8 символів"

> **Boilerplate-нотатка:** Кожен бізнес може налаштувати вимоги під свої потреби: додати вимоги до великих літер, цифр, спецсимволів, або підключити zxcvbn для оцінки сили пароля.

## Технічні нотатки

### Backend endpoint: перевірка email

Потрібен новий endpoint (або модифікація існуючого):

```
POST /api/auth/check-email
Body: { email: string }
Response: { hasPassword: boolean, isNewUser: boolean }
```

> **Безпека:** Цей endpoint розкриває чи існує email в базі. Це прийнятний trade-off для UX (такий самий підхід у Slack, Notion, GitHub). Rate limit per-IP обов'язковий (10 req/min). Для forgot password використовується окрема стратегія — однакова відповідь незалежно від існування email.

### Magic link context

Magic link має нести контекст (для якого сценарію він створений), щоб профіль знав який UI показувати:

- `purpose: 'login'` — Сценарій B (існуючий без пароля)
- `purpose: 'register'` — Сценарій C (новий юзер)
- `purpose: 'reset-password'` — Forgot password

Контекст зберігається в Redis разом з token.

### Зміни в User schema

Нове поле:

```
passwordHash: string | null  // bcrypt hash, null = немає пароля
```

### Зміни в Redis keys

Нові ключі:

```
login_attempts:{ip}:{email} → count (TTL = AUTH_LOGIN_ATTEMPTS_TTL_MIN)
check_email:{ip} → count (TTL = 60s)
magic_dedup:{email}:{purpose} → token (TTL = AUTH_MAGIC_LINK_DEDUP_SEC)
```

## Заділ на майбутнє

> Ці функції **не реалізуються зараз**, але архітектура має їх враховувати. Задокументовано для планування.

### 2FA (Two-Factor Authentication)

Точка розширення в auth flow: після успішної перевірки пароля (Сценарій A) додається перевірка `if (user.has2FA)`. Замість одразу видавати access token, backend повертає тимчасовий `2fa_token` (короткий TTL), який дає право лише на один endpoint — `POST /api/auth/verify-2fa`. Після успішної верифікації TOTP-коду видається звичайна пара токенів.

### Зміна email

Безпечний процес зміни основного ідентифікатора:
1. Юзер вводить новий email у профілі
2. На **новий** email відправляється magic link (`purpose: verify-new-email`)
3. Поки посилання не натиснуте — email в базі не змінюється
4. (Опціонально) На **старий** email відправляється повідомлення: "Ваш email було змінено. Якщо це були не ви — зверніться в підтримку"

### Активні сесії

Розділ "Активні сесії" у профілі: список пристроїв (User-Agent + IP, збережені в Redis разом з refresh token) + кнопка "Вийти з усіх інших пристроїв" (`revokeAllUserTokens()` з виключенням поточного jti).
