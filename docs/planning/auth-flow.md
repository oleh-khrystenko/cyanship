## Принципи

1. **Один email = один акаунт** — незалежно від провайдера (Google, magic link, пароль). Якщо юзер зареєструвався через Google і потім вводить той самий email — це той самий акаунт.
2. **Пароль — опціональний** — юзер завжди може входити через OAuth або magic link без пароля. Пароль — додатковий рівень зручності та безпеки.
3. **Progressive disclosure** — спочатку тільки поле email, далі контекстно показується або поле пароля, або повідомлення про magic link.
4. **Boilerplate-friendly** — ключові параметри (кількість спроб, довжина пароля, тощо) мають бути конфігурованими, а не захардкодженими.

## Сторінка авторизації

### UI елементи

```
┌─────────────────────────────────────────┐
│                                         │
│            Вхід до LucidKit             │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │      Увійти з Google              │  │
│  └───────────────────────────────────┘  │
│                                         │
│           ──── або ────                 │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │  Email                            │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │           Продовжити              │  │
│  └───────────────────────────────────┘  │
│                                         │
└─────────────────────────────────────────┘
```

### Поведінка

1. Юзер вводить email і натискає "Продовжити"
2. Показується прелоадер
3. Backend перевіряє: чи існує юзер і чи має пароль
4. В залежності від результату — один з трьох сценаріїв (див. нижче)

## Сценарії email-авторизації

### Сценарій A: Юзер існує + має пароль

**Тригер:** Backend повертає `{ hasPassword: true }`

**UI:** Під полем email з'являється поле пароля + кнопка "Увійти" + посилання "Забув пароль?"

```
┌─────────────────────────────────────────┐
│  ┌───────────────────────────────────┐  │
│  │  email@example.com           [✓]  │  │
│  └───────────────────────────────────┘  │
│  ┌───────────────────────────────────┐  │
│  │  Пароль                           │  │
│  └───────────────────────────────────┘  │
│              Забув пароль?              │
│  ┌───────────────────────────────────┐  │
│  │             Увійти                │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

**Після успішного входу:** Redirect на головну сторінку (`/`), НЕ на профіль.

---

### Сценарій B: Юзер існує + НЕ має пароля

**Тригер:** Backend повертає `{ hasPassword: false, isNewUser: false }`

**UI:** Прелоадер зникає, з'являється повідомлення:

```
┌─────────────────────────────────────────┐
│                                         │
│        Перевірте вашу пошту             │
│                                         │
│  Ми надіслали посилання для входу на    │
│  email@example.com                      │
│                                         │
│  Не отримали листа? Перевірте спам      │
│  або спробуйте ще раз.                  │
│                                         │
└─────────────────────────────────────────┘
```

**Email:** Лист з текстом, що відповідає ситуації — юзер вже зареєстрований, ми надсилаємо посилання для входу та можливість встановити пароль.

---

### Сценарій C: Юзер НЕ існує (новий)

**Тригер:** Backend повертає `{ hasPassword: false, isNewUser: true }`

**UI:** Ідентичний сценарію B (юзер бачить "Перевірте вашу пошту").

**Email:** Лист з текстом про реєстрацію — вітання, magic link для активації акаунту та заповнення профілю.

> **Важливо:** З точки зору UI, сценарії B та C виглядають однаково для юзера. Різниця тільки в тексті email та в стані сторінки профілю після переходу по magic link.

## Magic Link

### Генерація

- Token: 64-byte hex (256-bit), зберігається в Redis
- TTL: 15 хвилин
- Rate limit: 3 запити на email за 15 хвилин
- Одноразовий: після використання видаляється з Redis

### Повторний клік на magic link (вже використаний)

Коли юзер натискає на magic link вдруге (token вже видалений з Redis):

1. Backend не знаходить token → повертає помилку
2. Redirect на сторінку авторизації
3. Toast-нотифікація: "Посилання вже було використано. Введіть email для отримання нового"

**Обгрунтування:** Magic link одноразовий з міркувань безпеки. Якщо хтось перехопив посилання і юзер вже його використав — зловмисник не зможе увійти. Повідомлення дружнє і одразу пропонує рішення.

### Expired magic link

Коли юзер натискає на magic link після закінчення TTL:

1. Backend не знаходить token (видалений по TTL) → повертає помилку
2. Redirect на сторінку авторизації
3. Toast-нотифікація: "Час дії посилання закінчився. Спробуйте ще раз"

> **Примітка:** Технічно expired та reused magic link повертають однакову помилку (token не знайдено в Redis). Backend не розрізняє ці випадки. Можна використовувати єдине повідомлення: "Посилання недійсне або вже було використано. Введіть email для отримання нового".

## Forgot Password

### Флоу

1. Юзер на сторінці авторизації бачить поле пароля (Сценарій A)
2. Натискає "Забув пароль?"
3. Backend відправляє magic link на email юзера
4. Toast: "Посилання для зміни пароля надіслано на вашу пошту"
5. Юзер натискає magic link → потрапляє на сторінку профілю
6. На сторінці профілю поле пароля в режимі редагування (обов'язкове для заповнення)
7. Юзер встановлює новий пароль → зберігає

### Email

Окремий шаблон листа для "Забув пароль" — текст про те, що був запит на зміну пароля та magic link.

## Профіль після magic link

Після успішного переходу по magic link юзер автентифікується (отримує сесію) і потрапляє на сторінку профілю (protected route). Стан сторінки залежить від контексту:

### Новий юзер (Сценарій C → magic link)

Профіль **повністю в режимі редагування:**

| Поле | Стан | Обов'язкове | Примітка |
|------|------|-------------|----------|
| Ім'я | Редагування | Так | Потрібне для звернення в листах та на платформі |
| Прізвище | Редагування | Ні | — |
| Аватар | Редагування | Ні | Можливість завантажити |
| Пароль | Редагування | Ні | Рекомендація: "Для безпеки ваших даних ми рекомендуємо встановити пароль" |

```
┌─────────────────────────────────────────┐
│          Заповніть ваш профіль          │
│                                         │
│  Ім'я *                                 │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│                                         │
│  Прізвище                               │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│                                         │
│  Пароль (рекомендовано)                 │
│  ┌───────────────────────────────────┐  │
│  │                                   │  │
│  └───────────────────────────────────┘  │
│  Для безпеки ваших даних ми             │
│  рекомендуємо встановити пароль          │
│                                         │
│  ┌───────────────────────────────────┐  │
│  │           Зберегти                │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

### Існуючий юзер без пароля (Сценарій B → magic link)

Профіль **частково в режимі редагування:**

| Поле | Стан | Обов'язкове | Примітка |
|------|------|-------------|----------|
| Ім'я | Перегляд (не редагується) | — | Вже заповнене |
| Прізвище | Перегляд (не редагується) | — | Може бути порожнім |
| Аватар | Перегляд (не редагується) | — | — |
| Пароль | Редагування | Ні | Рекомендація: "Для безпеки ваших даних ми рекомендуємо встановити пароль" |

---

### Forgot password (існуючий юзер з паролем → magic link)

Профіль **частково в режимі редагування:**

| Поле | Стан | Обов'язкове | Примітка |
|------|------|-------------|----------|
| Ім'я | Перегляд | — | — |
| Прізвище | Перегляд | — | — |
| Аватар | Перегляд | — | — |
| Пароль | Редагування | Так | Юзер прийшов скинути пароль — поле обов'язкове |

## Управління паролем у профілі

На сторінці профілю (доступна з будь-якої точки додатку) є секція "Безпека":

### Юзер без пароля

- Показується кнопка "Встановити пароль"
- Поле: новий пароль
- Після збереження: пароль з'являється, наступний вхід через email покаже поле пароля

### Юзер з паролем

- Показується кнопка "Змінити пароль"
- Поля: поточний пароль → новий пароль
- Після збереження: пароль оновлюється

### Видалення пароля

- Показується кнопка "Видалити пароль" (або перемикач)
- Підтвердження: "Ви впевнені? Після видалення пароля вхід буде можливий тільки через OAuth або magic link"
- Після видалення: наступний вхід через email відправить magic link замість показу поля пароля

> **Примітка (boilerplate):** Можливість видалення пароля збережена для гнучкості. Конкретний бізнес може приховати цю опцію якщо вважає за потрібне.

## OAuth (Google та інші)

### Поточна реалізація: Google

1. Юзер натискає "Увійти з Google"
2. Redirect на Google consent screen
3. Google callback → `AuthService.handleGoogleAuth()`
4. Якщо юзера немає — створюється (email, profile.name, profile.avatar з Google)
5. Set `bid_refresh` cookie → redirect на callback page
6. Web: `refreshToken()` → `getMe()` → redirect на головну

### Майбутні провайдери

Facebook, Apple та інші додаються за тим самим патерном: кнопка на сторінці авторизації + Passport strategy + callback handler. Принцип "один email = один акаунт" зберігається.

## Account Linking

### Правило

Один email = один акаунт. Провайдер не створює окремий акаунт.

### Сценарії

| Перша реєстрація | Друга дія | Результат |
|------------------|-----------|-----------|
| Google (user@gmail.com) | Email вхід user@gmail.com | Той самий акаунт, magic link для входу (пароля немає) |
| Magic link (user@gmail.com) | Google OAuth з user@gmail.com | Той самий акаунт, Google провайдер додається до акаунту |
| Email + пароль (user@gmail.com) | Google OAuth з user@gmail.com | Той самий акаунт, Google провайдер додається, пароль зберігається |

## Захист від brute force

### Невірний пароль

- **Максимум спроб:** конфігурована змінна (default: `100`)
- **Блокування:** після досягнення ліміту — тимчасовий блок для цього email
- **Тривалість блокування:** конфігурована змінна (default: `15 хвилин`)
- **Повідомлення юзеру:** "Забагато невдалих спроб. Спробуйте через 15 хвилин або скористайтесь посиланням 'Забув пароль?'"
- **Зберігання:** Redis (key: `login_attempts:{email}`, TTL = тривалість блокування)

### Конфігурація (env або config)

```
AUTH_MAX_LOGIN_ATTEMPTS=100
AUTH_LOGIN_BLOCK_DURATION_MIN=15
AUTH_PASSWORD_MIN_LENGTH=8
AUTH_MAGIC_LINK_TTL_MIN=15
AUTH_MAGIC_LINK_RATE_LIMIT=3
AUTH_MAGIC_LINK_RATE_WINDOW_MIN=15
```

> **Boilerplate-нотатка:** Значення `100` для `AUTH_MAX_LOGIN_ATTEMPTS` обране як баланс — занадто мале число дратує легітимних юзерів, `100` спроб зупинить будь-який скрипт/бот, при цьому жодна реальна людина не помилиться 100 разів.

## Вимоги до пароля

### Мінімальні (default)

- Мінімальна довжина: конфігурована змінна (default: `8` символів)
- Без додаткових вимог до складності

### Валідація

- Zod-схема в `packages/types` — single source of truth
- Перевірка на frontend (UX) та backend (безпека)
- Повідомлення: "Пароль має містити щонайменше 8 символів"

> **Boilerplate-нотатка:** Кожен бізнес може налаштувати вимоги під свої потреби: додати вимоги до великих літер, цифр, спецсимволів, або підключити zxcvbn для оцінки сили пароля.

## Error States

| Ситуація | UI поведінка | Toast/повідомлення |
|----------|-------------|-------------------|
| Невірний пароль | Поле пароля підсвічується червоним | "Невірний пароль" |
| Brute force блок | Поле пароля заблоковане | "Забагато невдалих спроб. Спробуйте через {N} хвилин або скористайтесь посиланням 'Забув пароль?'" |
| Expired magic link | Redirect на сторінку авторизації | "Час дії посилання закінчився. Спробуйте ще раз" |
| Використаний magic link | Redirect на сторінку авторизації | "Посилання недійсне або вже було використано. Введіть email для отримання нового" |
| Rate limit magic link | Залишається на поточному стані | "Забагато запитів. Спробуйте через кілька хвилин" |
| Google OAuth помилка | Redirect на сторінку авторизації | "Не вдалося увійти через Google. Спробуйте ще раз" |
| Невалідний email | Поле email підсвічується червоним | "Введіть коректну email-адресу" |
| Сервер недоступний | Прелоадер зникає | "Сервіс тимчасово недоступний. Спробуйте пізніше" |

## Технічні нотатки

### Backend endpoint: перевірка email

Потрібен новий endpoint (або модифікація існуючого):

```
POST /api/auth/check-email
Body: { email: string }
Response: { hasPassword: boolean, isNewUser: boolean }
```

> **Безпека:** Цей endpoint розкриває чи існує email в базі. Це прийнятний trade-off для UX (такий самий підхід у Slack, Notion, GitHub). Rate limit обов'язковий.

### Magic link context

Magic link має нести контекст (для якого сценарію він створений), щоб профіль знав який UI показувати:

- `purpose: 'login'` — Сценарій B (існуючий без пароля)
- `purpose: 'register'` — Сценарій C (новий юзер)
- `purpose: 'reset-password'` — Forgot password

Контекст зберігається в Redis разом з token.

### Зміни в User schema

Нове поле:

```
passwordHash: string | null  // bcrypt hash, null = немає пароля
```

### Зміни в Redis keys

Новий ключ для login attempts:

```
login_attempts:{email} → count (TTL = AUTH_LOGIN_BLOCK_DURATION_MIN)
```
