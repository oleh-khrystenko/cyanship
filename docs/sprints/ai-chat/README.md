# AI Chat — Sprint Plan

> **Ціль:** Додати на дашборд живий чат з AI, який демонструє повну інтеграцію штучного інтелекту з існуючою системою біллінгу та балансу executions. Лід бачить: написав повідомлення → AI відповів у реальному часі → баланс зменшився → транзакція зʼявилась в історії.

---

## Що будуємо і навіщо

CyanShip — це демо-бойлерплейт для стартапів. Наша задача — показати потенційному клієнту, що платформа вже містить готову інтеграцію з AI. Клієнт відкриває дашборд, бачить чат, пише повідомлення, отримує відповідь від справжнього AI з ефектом "друкування" (текст зʼявляється по словах), і одразу бачить як зменшився баланс. Це живий proof що біллінг і AI працюють разом.

---

## Ключові рішення

### 1. Який AI використовуємо

Ми **не привʼязуємось до одного постачальника**. Система побудована як конструктор:
- За замовчуванням підключений **Anthropic** (модель Claude Haiku — найдешевша, але справжній AI)
- Клієнт може замінити на OpenAI, Google Gemini чи будь-що інше — потрібно лише написати один файл-адаптер і змінити налаштування
- Це той самий підхід, який ми вже використовуємо для Stripe — де Stripe можна замінити на інший платіжний сервіс

**Вартість**: один запит до Haiku коштує приблизно $0.0004 (менше ніж 1/10 цента). При ліміті 5 запитів на користувача — максимум $0.002 на одного ліда.

### 2. Захист від зловживань (3 рівні)

Оскільки кожен запит до AI коштує реальні гроші, ми захищаємось трьома способами:

| Рівень | Що перевіряє | Ліміт | Що бачить користувач |
|--------|-------------|-------|---------------------|
| **IP-адреса** | Не більше 5 запитів з однієї IP за 24 години | 5 / 24 год | "Ліміт вичерпано, спробуйте завтра" |
| **Акаунт** | Не більше 5 запитів з одного акаунту за 24 години | 5 / 24 год | "Ліміт вичерпано, спробуйте завтра" |
| **Баланс** | Перевірка що на рахунку достатньо executions | 200 exec/запит | "Недостатньо executions" |

Навіщо і IP, і акаунт? Якщо хтось створить 10 акаунтів з однієї IP — IP-ліміт його зупинить. Якщо хтось спробує з різних IP одним акаунтом — зупинить ліміт акаунту.

Додатково: AI обмежений у довжині відповіді (максимум ~150 слів), тому навіть при зловживанні вартість мізерна.

### 3. Вартість одного запиту

Кожне повідомлення в чаті списує **200 executions** з балансу користувача. Це:
- Достатньо помітно для демонстрації (баланс реально зменшується)
- Не забагато — starter план (~5,000 executions) дозволяє 25 повідомлень
- При ліміті 5 повідомлень на акаунт — максимум 1,000 executions

### 4. Ефект "друкування" (Streaming)

Відповідь AI зʼявляється не одразу цілком, а по словах — як ніби хтось друкує в реальному часі. Це стандартна поведінка ChatGPT, яку всі очікують. Технічно це називається Server-Sent Events (SSE) — браузер отримує відповідь по частинах по мірі того як AI її генерує.

### 5. Де знаходиться чат

На дашборді — як нова секція між spend-кнопками та історією транзакцій. Це підсилює демо: користувач бачить чат, баланс і транзакції на одному екрані.

## Що НЕ робимо (і чому)

| Рішення | Причина |
|---------|---------|
| Не зберігаємо історію чатів у базі | Це демо, а не продукт. Історія живе поки вкладка відкрита |
| Не даємо вибирати модель AI | Одна модель — простіше, дешевше, менше точок відмови |
| Не підтримуємо файли чи зображення | Text-only достатньо для демонстрації інтеграції |
| Не даємо AI "памʼять" між повідомленнями | Кожне повідомлення незалежне — дешевше і безпечніше |
| Не робимо admin-панель для AI | Непотрібний оверінжиніринг для демо |

## Фази реалізації

### Фаза 1 — Shared Types (пакет `packages/types`)

**Що**: Додаємо нову дію `ai_chat` до системи executions та описуємо контракти для AI чату.

**Результат**: Обидва додатки (API та Web) знають про нову дію та формат даних для чату.

**Деталі**:
- Нова дія витрат: `ai_chat` з вартістю 200 executions
- Контракт запиту: повідомлення користувача (текст, макс. 500 символів)
- Контракт відповіді: формати SSE-подій (token, error, done)
- Конфігурація обмежень: максимум 5 запитів на акаунт, 5 на IP, 24-годинне вікно

### Фаза 2 — Backend: AI Module

**Що**: Новий модуль на сервері який приймає повідомлення, перевіряє ліміти, списує баланс та стрімить відповідь від AI.

**Результат**: Робочий API-ендпоінт `POST /ai/chat` який повертає streaming-відповідь.

**Деталі**:

**Крок 2.1 — Інтерфейс AI-провайдера**
- Створюємо "контракт" для будь-якого AI-провайдера (як ми зробили для Stripe)
- Контракт описує одну дію: "надішли повідомлення — отримай потік відповіді"
- Будь-який провайдер (Anthropic, OpenAI, Gemini) має реалізувати цей контракт

**Крок 2.2 — Реалізація Anthropic-провайдера**
- Підключення до Anthropic API через офіційний SDK
- Модель: Claude Haiku (найдешевша, але справжня)
- Системний промпт: "Відповідай коротко, 2-3 речення максимум"
- Обмеження відповіді: ~150 слів (жорсткий ліміт на рівні API)

**Крок 2.3 — Guard обмежень (IP + акаунт)**
- Перед кожним запитом перевіряємо лічильники в Redis
- Якщо IP або акаунт перевищили 5 запитів за 24 години — відмова з кодом 429
- Лічильники автоматично скидаються через 24 години

**Крок 2.4 — Сервіс та контролер**
- Контролер приймає повідомлення, запускає pipeline:
  1. Guard перевіряє ліміти (IP + акаунт)
  2. Списуємо 200 executions з балансу (існуючий механізм)
  3. Якщо недостатньо — повертаємо помилку
  4. Відправляємо запит до AI
  5. Стрімимо відповідь по словах через SSE
  6. В кінці відправляємо подію "done" з новим балансом

**Крок 2.5 — Реєстрація модуля**
- Підключаємо AI-модуль до головного AppModule
- Додаємо нову env-змінну `ANTHROPIC_API_KEY`

### Фаза 3 — Frontend: Chat UI

**Що**: Компонент чату на дашборді з streaming-відображенням відповідей.

**Результат**: Користувач бачить чат, пише повідомлення, отримує відповідь з ефектом "друкування", баланс оновлюється.

**Деталі**:

**Крок 3.1 — API-функція для чату**
- Функція яка відправляє повідомлення на сервер
- Читає streaming-відповідь по частинах (через Fetch API, не Axios — бо Axios не підтримує SSE)
- Передає кожне слово callback-функції для відображення

**Крок 3.2 — Компонент чату**
- Мінімалістичний UI: поле вводу + список повідомлень
- Повідомлення користувача — справа (як в месенджерах)
- Відповідь AI — зліва, текст зʼявляється по словах
- Під час генерації: кнопка "Відправити" заблокована, показується індикатор
- Після завершення: баланс на дашборді оновлюється автоматично
- Обробка помилок: якщо ліміт вичерпано або недостатньо executions — toast-повідомлення

**Крок 3.3 — Інтеграція в дашборд**
- Чат зʼявляється як нова секція на сторінці дашборду
- Розміщення: між spend-кнопками та історією транзакцій
- Після кожного повідомлення: оновлюємо баланс у header + оновлюємо список транзакцій

### Фаза 4 — Translations (i18n)

**Що**: Тексти для чату українською та англійською.

**Результат**: Весь UI чату локалізований.

**Деталі**:
- Заголовок секції, placeholder, кнопка відправки
- Повідомлення помилок: ліміт вичерпано, недостатньо executions, AI недоступний
- Інформаційний текст: вартість запиту, залишок лімітів
- Нова дія `ai_chat` в історії транзакцій

### Фаза 5 — Environment & Configuration

**Що**: Нові налаштування сервера.

**Результат**: AI модуль правильно налаштований і валідований при старті.

**Деталі**:
- Нова обовʼязкова змінна: `ANTHROPIC_API_KEY`
- Нові опціональні змінні з дефолтами: `AI_CHAT_MAX_TOKENS` (150), `AI_CHAT_IP_LIMIT` (5), `AI_CHAT_USER_LIMIT` (5)
- Оновлення `.env.example` з прикладами
- Оновлення `test-setup.ts` з fallback-значеннями для тестів

### Фаза 6 — Testing

**Що**: Тести для всіх нових компонентів.

**Результат**: Впевненість що все працює коректно.

**Деталі**:
- **Unit-тести API**: guard перевіряє ліміти, сервіс списує баланс, провайдер стрімить відповідь
- **E2E-тест API**: повний цикл запит → списання → streaming → done event
- **Unit-тести Web**: чат-компонент рендериться, відправляє повідомлення, показує streaming
- Mock AI-провайдера для тестів (не робимо реальних запитів до Anthropic в CI)

## Ризики та мітігація

| Ризик | Наслідок | Як запобігаємо |
|-------|---------|----------------|
| Хтось намагається "зламати" AI через хитрі промпти | AI ігнорує інструкції, видає довгу відповідь | Жорсткий ліміт на довжину відповіді (не в промпті, а на рівні API — AI фізично не може написати більше) |
| Anthropic API не працює | Чат не відповідає | Показуємо зрозуміле повідомлення "AI тимчасово недоступний". Executions НЕ списуються якщо AI не відповів |
| API ключ потрапив у код | Витрати на чужому ключі | Ключ тільки на сервері, фронт ніколи його не бачить |
| Redis не працює | Ліміти не перевіряються | Блокуємо всі AI-запити поки Redis не повернеться (fail-closed — краще заблокувати ніж пропустити) |
| Вартість виросла | Непередбачувані витрати | 3 рівні захисту + короткі відповіді = максимум ~$0.002 на користувача |
